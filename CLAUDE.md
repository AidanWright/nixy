# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Purpose

The purpose of this project is to provide a clean nix configuration for multiple hosts including desktops, laptops, and servers. The hosts can be either NixOS or Nix-Darwin on any architecture. The goal is to make developing, testing, and deploying hosts as easy as possible.

## Rules

There is a list of non-negotiables that (in order of importance) appears below:

1. Native solutions are always preferred. Refer to the list of tooling this project uses [below](#tools) to find a *(mostly)* comprehensive list.
Before any custom solutions are created, always check the documentation and the implementation of the already used technology. Often, there is a native way to do things.

2. The project should be accessible to __anyone__. All code written must be runnable by anyone under any particular set of circumstances.
That means values should not be hard coded and room for expansion must be built in. That also means that if additional packages are needed or additional setup is required by the user,
it must be well documented. We never assume an architecture or operating system. If we clone this project, it should be reasonable to get every aspect working with minimal effort.

3. Modules are single-purpose. We use the dendretic pattern on purpose. A file should do __one__ thing, never more. A file that configures SSH on Darwin host `A`, shouldn't also configure the desktop envoirenment--and certainly not on host `B`.

4. Commenting is almost always wrong. The code itself should be the documentation. The only reason for a comment is to explain why a particular implementation choice was made over another, or explain an otherwise stange implementation.
We never just write what the code is doing. That is always wrong.

5. Following bullet 4, code should be readable. Variable names should be descriptive! Any logic should be broken out into a separate function. Any repeated logic should be factored out. Likewise, functions are single-purpose.

6. Readers’ time and attention are limited. Respect them. Follow the evidence-based [plain language guidelines](https://digital.gov/guides/plain-language).

- Don’t use jargon. Readers may not be familiar with particular technical terms.
- Don’t use long, complicated words if there are shorter, simpler words that convey the same meaning.

Use the imperative voice when giving instructions. For example, write:
> Add the python310 package to buildInputs.

Don’t use a conversational tone, as it distracts from the content. For example, don’t write:
> Going forward, let’s now add the python310 package to buildInputs as we have seen in the previous tutorial.

## Tools

- Nix:
    - [Nix](https://nix.dev/): the underlying dependancy of our project. Used in all parts.
    - [Lix](https://docs.lix.systems/manual/lix/stable/): An opinionated alternative implementation of nix. We only use with darwin systems currently. 
    - [Determinate Nix](https://docs.determinate.systems/): Another alternative implementation of nix. Has lazy trees and native linux builders on MacOS. Our default choice for darwin systems.

- File-Structure:
    - We use the dendretic pattern and stick to the design priciples layed out by [dendritic design with flake parts](https://github.com/Doc-Steve/dendritic-design-with-flake-parts/wiki). The layout described in this documentation should be continually referenced while developing this project.
    - See: [module layout](#module-layout)

- [flake-parts](https://flake.parts/): The heart of the dendretic pattern. It reduces the proliferation of custom Nix glue code, similar to what the module system has done for NixOS configurations. flake-parts contains many optional modules and should be searched for potential native solutions.

- [import-tree](https://import-tree.denful.dev/): Keeps imports close to where they are used. The main `flake.nix` should never be modified. Instead, it is regenerated by `nix run .#write-flake` after including a new import with import-tree.

- Virtualization:
    - [Tart](https://tart.run/): Manages creation of MacOS guests (MacOS Apple Silicon host only).
    - [nixos-shell](https://github.com/mic92/nixos-shell): Manages creation of Linux guests.

- [treefmt](): Formats all code. Must be run before commiting. Also creates headers at top of file that contain the information of what the file does. This is the only place in the file where a description of *what* the code does should be. Following the [rules](#rules), every other comment in the file must be an description of *why* the code does. These headers should include any commands exposed by the file and give a general overview without being verbose. Concise is always better.

## Commands

```bash
# Format all code
nix fmt

# Regenerate flake.nix from module declarations (auto-generated — do not edit directly)
nix run .#write-flake

# Show all flake outputs
nix flake show

# Build and push the shared base image (Lix + nix-darwin) to the OCI registry
nix run --impure .#tart-base-image

# Apply a host's config to a VM from the base image and run it
nix run --impure .#darwinConfigurations.<HOSTNAME>.config.system.build.tart-vm
```

### Module layout

```

├── CLAUDE.md                                   # This file
├── flake.lock                                 
├── flake.nix                                   # Defines nix inputs/outputs -- DO NOT EDIT DIRECTLY
└── modules/                                    # All *.nix files within will be automatically imported
    ├── hosts/                                  # Each host gets it's own sub-directory
    │   └── hostname/                            
    │       ├── configuration.nix               # Where the host is "configured" (i.e. all aspects are imported)
    │       └── flake-parts.nix                 # Where host is enabled
    ├── nix/                                    # For all nix specific wiring/helpers
    │   ├── dendritic.nix                       # The basic set-up for the dendretic system 
    │   ├── flake-parts/                        # Helper functions for the flake-parts tool
    │   ├── overlays/
    │   └── tools/                              # Any tools that the project uses like formatting.
    ├── services/                               # Services like printing or SSH
    ├── system/                                 # Aspects that define a system. e.g. desktop configurations, basic packages, impermanance, nix variants, etc.
    │   ├── determinate.nix
    │   ├── impermanence.nix
    │   ├── virtualization/
    │   │   ├── cirrus.yml                      # Cirrus CI task template for the tart VM runner
    │   │   ├── nixos-shell.nix                 # nixos-shell runner for linux guests
    │   │   ├── tart-base-image.nix             # builds and pushes the shared Tart base image
    │   │   └── tart-vm.nix                     # per-host darwin module for running config in a VM
    └── users/                                  # Home-manager user settings
```

### Adding a new host

1. Create `modules/hosts/<name>/configuration.nix` (imports the desired system modules)
2. Create `modules/hosts/<name>/flake-parts.nix` (calls `flake.lib.mkDarwin` or `flake.lib.mkNixos`)

The `flake-parts.nix` pattern used by macbook is the reference implementation.

### Code formatting

Treefmt is configured in `modules/nix/tools/treefmt.nix`. It runs `nixfmt` on `.nix` files and a custom Python header formatter (`tools/header.py`) that manages file-path comment headers. Run with `nix fmt`.
